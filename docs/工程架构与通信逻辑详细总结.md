## 2. 系统整体架构

当前项目是一个 Qt Widgets 工业工站软件，核心对象大致如下：

```text
MFormJLStation 主界面
    |
    |-- MFormWorkStep        品番/频道/扫码规则/ATEQ模式设置
    |-- MForm_PrintSetting   OK/NG标签打印参数与BarTender打印
    |-- MForm_ScanCode       Honeywell扫码枪串口
    |-- MFormATEQ            ATEQ串口设置界面
    |-- COMLeakHunting       ATEQ1串口接收解析
    |-- COMLeakHunting_2     ATEQ2串口接收解析
    |-- MForm_ModeBusRtu     PLC监视窗口
    |-- mModbusRtu           PLC读写封装
    |-- mWorkTask            检测流程工作线程/状态机
    |-- MFormMysql           数据库界面
    |-- MMYSQLDatas          数据库后台线程
```

主流程由 `mWorkTask` 驱动，主界面 `MFormJLStation` 主要负责显示、接收外设信号、转发用户操作和更新 UI。

PLC 通信由 `mModbusRtu` 封装，当前实际通过三菱 ActiveX 控件 `ActUtlType` 的 `GetDevice/SetDevice` 完成读写，而不是传统 Modbus RTU 帧协议。

## 3. 主要模块职责

### 3.1 主界面 MFormJLStation

文件：

- `MFormJLStation.cpp`
- `MFormJLStation.h`
- `MFormJLStation.ui`

主要职责：

- 创建和连接各子模块。
- 管理主界面 UI 显示。
- 接收扫码枪条码信号。
- 接收 ATEQ 串口解析后的压力、泄漏值。
- 将条码和当前界面快照交给 `mWorkTask`。
- 提供软件复位按钮。
- 切换频道时同步 `MFormWorkStep`、`MForm_PrintSetting` 和 PLC 品番字。
- 根据工作线程信号更新灯、文本、检测时间、条码等界面内容。

关键函数：

| 函数 | 作用 |
| --- | --- |
| `MFormJLStation::MFormJLStation()` | 初始化主界面、创建子模块、连接信号、启动线程和定时器 |
| `slot_onBarcodeReceived()` | 接收扫码枪条码，校验频道，生成快照并提交给工作线程 |
| `slot_updateAteqDisplay()` | 接收 ATEQ 解析后的压力/泄漏值，按本轮 ATEQ 模式更新对应 UI |
| `on_CBT_Channel_currentTextChanged()` | 主界面频道切换，刷新品番参数、打印参数并写 PLC 品番字 |
| `on_BTN_Reset_clicked()` | 软件复位按钮，向工作线程发复位请求并清 UI |
| `applyManualResetUi()` | 软件复位和实体复位共用的 UI 清空效果 |
| `triggerScanCode()` | 触发扫码枪发码 |
| `slot_init_Display()` | 清空检漏结果、压力、泄漏值、条码等显示 |

### 3.2 工作线程 mWorkTask

文件：

- `mWorkTask.cpp`
- `mWorkTask.h`

主要职责：

- 运行检测主状态机。
- 接收条码并缓存为 `pendingBarcode`。
- 将 `pendingBarcode` 转为 `currentBarcode`，进入本轮检测。
- 控制扫码前预复位、PLC 启动检漏、等待 PLC 结果、等待 ATEQ 数据、排气收尾、保存和打印。
- 处理软件复位、实体复位、防错复位。
- 维护本轮 ATEQ 模式、ATEQ 数据是否已收到、PLC 总结果是否已锁定等状态。

关键状态：

| 状态/字段 | 说明 |
| --- | --- |
| `b_gameover` | 当前工位是否空闲/本轮是否结束 |
| `pendingBarcode` | 已收到但尚未进入流程的条码 |
| `currentBarcode` | 当前正在检测的条码 |
| `pendingSnapshot/currentSnapshot` | 条码进入流程时的 UI 快照 |
| `cycleAteqEnable` | 本轮锁定的 ATEQ 模式，避免检测中切品番影响判断 |
| `plcLeakResultLatched` | PLC 总结果是否已经锁定 |
| `plcOverallResultOk` | PLC 总结果，`true=OK`，`false=NG` |
| `ateq1DataReceived` | 本轮 ATEQ1 串口数据是否收到 |
| `ateq2DataReceived` | 本轮 ATEQ2 串口数据是否收到 |
| `preResetAction` | 扫码前预复位动作状态 |
| `manualResetRequested` | UI/实体复位请求标志 |

### 3.3 PLC 通信 mModbusRtu

文件：

- `ModbusRtu/mModbusRtu.cpp`
- `ModbusRtu/mModbusRtu.h`

主要职责：

- 封装 PLC 读写。
- 维护本地镜像字：`D1900`、`D1901`、`D1902`、`D1903`、`D90`、`D91`、`N10`。
- 按当前品番和 ATEQ 模式生成 PLC 输出。
- 将 PLC 输入映射回旧框架 `recDatasDi`。
- 产生软件心跳、软件准备、ATEQ 使能、启动检漏、全部复位等信号。

当前底层通信方式：

```cpp
Act_Type_PLC_1->GetDevice(device, value)
Act_Type_PLC_1->SetDevice(device, value)
```

跨线程访问 ActiveX 控件时使用：

```cpp
QMetaObject::invokeMethod(..., Qt::BlockingQueuedConnection)
```

这样可以避免工作线程直接调用 UI/ActiveX 所在线程对象。

### 3.4 扫码枪 MForm_ScanCode

文件：

- `Honeywell/MForm_ScanCode.cpp`
- `Honeywell/MForm_ScanCode.h`

主要职责：

- 打开扫码枪串口。
- 发送扫码触发命令。
- 接收扫码枪返回数据。
- 使用 50ms 单次定时器合并串口分段。
- 将完整条码通过信号发给主界面。

扫码触发命令：

```text
16 54 0D
```

扫码返回处理：

```text
readyRead -> readAll -> 累加缓存 -> 50ms 后 slot_readAll -> trimmed -> sign_barcodeReceived
```

### 3.5 ATEQ 串口模块

文件：

- `MATEQ/combase.cpp`
- `MATEQ/combase.h`
- `MATEQ/MFormATEQ.cpp`
- `MATEQ/MFormATEQ.h`

主要职责：

- 打开 ATEQ1/ATEQ2 串口。
- 接收 ATEQ 返回字符串。
- 使用 50ms 单次定时器合并串口分段。
- 解析压力、OK/NG、泄漏值。
- 通过 `sign_ATEQ_Display(device, pressure, leakage)` 发给主界面。

ATEQ 字符串示例：

```text
<01>:0.300 MPa:(OK):0.075 ml/min
<01>:0.300 MPa:(NG):0.300 ml/min
```

解析逻辑：

1. 用冒号 `:` 切分。
2. 查找包含 `OK` 或 `NG` 的片段。
3. 结果片段前一项为压力。
4. 结果片段后一项为泄漏值。
5. 结果本身去除括号。

当前 ATEQ 串口只将压力和泄漏值显示给界面，OK/NG 最终判定不直接由 ATEQ 串口文本决定，而由 PLC 总结果决定。

### 3.6 品番/频道配置 MFormWorkStep

文件：

- `MFormWorkStep.cpp`
- `MFormWorkStep.h`
- `MFormWorkStep.ui`

主要职责：

- 保存和加载频道参数。
- 保存扫码规则。
- 保存 ATEQ 模式。
- 保存 PLC 品番字配置。
- `Use` 按钮应用当前频道参数到主界面和运行参数。

关键配置文件：

```text
setting/channel.ini
```

重要字段：

| 字段 | 说明 |
| --- | --- |
| `product_name` | 当前频道产品显示名 |
| `cbt_ateq` | ATEQ 模式，通常 `1=ATEQ1`，`2=ATEQ2`，`3=双ATEQ` |
| `code1` | 条码固定字段规则 |
| `CodeLength_1` | 条码长度 |
| `CodeStation_1` | 条码固定字段起始位置 |
| `plc_product_ateq1` | 写给 PLC 的 ATEQ1 品番字 |
| `plc_product_ateq2` | 写给 PLC 的 ATEQ2 品番字 |

频道切换防重入：

- 主界面切频道时用 `QSignalBlocker` 阻断 `WorkStep::Channel` 自动触发。
- 同步打印设置频道时也用 `QSignalBlocker`。
- 阻断信号后，通过 `QMetaObject::invokeMethod` 手动调用打印设置加载函数，确保打印参数仍刷新一次。
- `Use` 按钮如果主界面已经是目标频道，不重复触发主界面频道切换。

### 3.7 打印模块 MForm_PrintSetting

文件：

- `MLabel/MForm_PrintSetting.cpp`
- `MLabel/MForm_PrintSetting.h`
- `MLabel/MForm_PrintSetting.ui`

主要职责：

- 管理 OK/NG 打印机名称。
- 管理各频道标签参数和流水号。
- 使用 BarTender ActiveX 打印 `.btw` 标签模板。
- 生成标签数据文本文件。

打印模板：

| 类型 | 模板 | 数据文件 |
| --- | --- | --- |
| OK 标签 | `label/channel_%1.btw` | `label/channel_%1.txt` |
| NG 标签 | `label/NG.btw` | `label/NG.txt` |

OK 标签打印：

```text
slot_Printe_Label_OK(channel)
    -> 加载频道打印参数
    -> 流水号 +1
    -> 写 channel_%1.txt
    -> 打开 channel_%1.btw
    -> BarTender PrintOut
```

NG 标签打印：

```text
slot_Printe_Label_NG(channel, leakValue)
    -> 加载频道打印参数
    -> 写 NG.txt
    -> 打开 NG.btw
    -> BarTender PrintOut
```

连接方式：

```cpp

```

检漏数据表：

```cpp
MYSQL_TABLE_JL_Datas
```

插入字段包括：

- 产品型号
- 设备号
- 频道
- 条码
- 批次

## 4. PLC 寄存器和位定义

### 4.1 软件写 PLC

| 寄存器/位 | 方向 | 含义 | 当前代码行为 |
| --- | --- | --- | --- |
| `D1900.0` | 软件 -> PLC | 心跳 | `slot_WriteCoils()` 周期翻转，约 1s 一次 |
| `D1900.1` | 软件 -> PLC | 软件准备完成 | 固定置 1 |
| `D1900.3` | 软件 -> PLC | ATEQ1 使能 | ATEQ 模式为 1 或 3 时置 1 |
| `D1900.4` | 软件 -> PLC | 全部复位脉冲 | `requestAllReset()` 置 3 个周期，约 300ms |
| `D1900.5` | 软件 -> PLC | 启动检漏脉冲 | `requestLeakStart()` 置 3 个周期，约 300ms |
| `D1900.9` | 软件 -> PLC | ATEQ2 使能 | ATEQ 模式为 2 或 3 时置 1 |
| `D1900.10` | 软件 -> PLC | 当前固定清 0 | 保留/未使用 |
| `D1902` | 软件 -> PLC | ATEQ1 目标品番字 | 切频道或扫码通过后写入 |
| `D1903` | 软件 -> PLC | ATEQ2 目标品番字 | 切频道或扫码通过后写入 |

### 4.2 PLC 给软件

| 寄存器/位 | 方向 | 含义 | 当前代码用途 |
| --- | --- | --- | --- |
| `D1901.0` | PLC -> 软件 | 工装到位 | 映射到旧输入，不是当前主流程核心 |
| `D1901.1` | PLC -> 软件 | 整体检测中/扫码触发相关状态 | 当前用其上升沿触发扫码前预复位和扫码 |
| `D1901.2` | PLC -> 软件 | PLC 总结果有效 | `JudgeResult` 必须等它为 1 才锁定结果 |
| `D1901.3` | PLC -> 软件 | PLC 总结果 | 最终整件 OK/NG 来源，1=OK，0=NG |
| `D1901.5` | PLC -> 软件 | ATEQ1 检测中 | 空闲判断辅助 |
| `D1901.6` | PLC -> 软件 | ATEQ1 单路有结果 | 仅用于显示 ATEQ1 单路 OK/NG |
| `D1901.7` | PLC -> 软件 | ATEQ1 单路结果 | 仅在 `D1901.6=1` 时用于显示 |
| `D1901.8` | PLC -> 软件 | ATEQ2 检测中 | 空闲判断辅助 |
| `D1901.9` | PLC -> 软件 | ATEQ2 单路有结果 | 仅用于显示 ATEQ2 单路 OK/NG |
| `D1901.10` | PLC -> 软件 | ATEQ2 单路结果 | 仅在 `D1901.9=1` 时用于显示 |
| `D1901.11` / `D1901.B` | PLC -> 软件 | 实体复位锁存请求 | 检测到后触发软件复位；软件用 `SetDevice("D1901.B", 0)` 清锁存 |
| `D90` | PLC -> 软件 | ATEQ1 当前品番反馈 | 与目标品番比对 |
| `D91` | PLC -> 软件 | ATEQ2 当前品番反馈 | 与目标品番比对 |
| `N10` | PLC -> 软件 | 物理/历史复位信号 | 与 `D1901.11` 一起作为实体复位来源 |

## 5. 线程和信号关系

### 5.1 UI 线程

以下对象主要工作在 UI 线程：

- `MFormJLStation`
- `MFormWorkStep`
- `MForm_PrintSetting`
- `MForm_ScanCode`
- `MFormMysql` 界面
- `MForm_ModeBusRtu`
- `MFormATEQ` 界面

### 5.2 工作线程

`mWorkTask` 被移动到自己的 `QThread` 中，主循环在 `WorkTask()` 中运行。

它不应直接操作 UI，而是通过信号：

- `sign_setled`
- `sign_set_text`
- `Sign_SetlineEdit`
- `sign_init_Display`
- `sign_applyManualResetUi`
- `sign_Printe_Label_OK`
- `sign_Printe_Label_NG`
- `sign_set_leakvalue`

### 5.3 ATEQ 串口线程

`mComThread` 启动后初始化 ATEQ1/ATEQ2 串口对象。ATEQ 返回后发信号给主界面：

```cpp
sign_ATEQ_Display(device, pressure, leakage)
```

主界面接收后调用：

```cpp
m_WorkTask->acceptAteqData(device, pressure, leakage)
```

若当前本轮允许该 ATEQ 数据，则更新对应 UI 的压力和泄漏值。

### 5.4 数据库线程

`MMYSQLDatas` 运行在独立线程中，负责数据库连接和插入。主流程通过信号：

```cpp
sign_update_JLDatas(mst_mysqlDatas)
```

将数据交给数据库线程。

## 6. 一件产品的完整流程

### 6.1 等待启动/扫码

工作线程循环中持续检查 PLC 和内部状态。当前扫码触发来源包括：

- PLC `D1901.1` 上升沿。
- 主界面启动按钮。
- 扫码枪直接扫码返回条码。

对于 PLC 触发扫码，流程会先进行扫码前预复位：

```text
D1901.1 上升沿
    -> requestAllReset()
    -> 等待 500ms
    -> triggerScanCode()
```

目的：

- 清上一轮 PLC 残留状态。
- 保持上一轮 OK/NG 灯在结果阶段不被立即清掉。
- 在下一轮开始时再清 PLC 状态。

### 6.2 扫码返回

扫码枪返回条码后：

```text
MForm_ScanCode::sign_barcodeReceived
    -> MFormJLStation::slot_onBarcodeReceived
    -> mWorkTask::acceptBarcode
    -> pendingBarcode
```

`acceptBarcode()` 会拒绝以下情况：

- 条码为空。
- 工位未空闲。
- 已存在待处理条码。
- 已存在当前条码。
- 急停状态。

### 6.3 条码进入检测流程

工作线程取出 `pendingBarcode` 后：

```text
pendingBarcode -> currentBarcode
pendingSnapshot -> currentSnapshot
resetLeakCycleState()
v_Step_WorkModel = Normal
v_Step_JL = WaitGetCode
```

本轮关键状态会初始化：

- `cycleAteqEnable`
- `plcLeakResultLatched=false`
- `plcOverallResultOk=true`
- `ateq1DataReceived=false`
- `ateq2DataReceived=false`
- `allResetRequested=false`
- `leakStartRequested=false`

### 6.4 条码规则验证

在 `Step_JL()` 的 `WaitGetCode` 阶段：

- 回填主界面条码。
- 校验固定字段。
- 校验条码长度。
- 清 ATEQ 接收状态。
- 根据当前 ATEQ 模式写品番字。

品番写入：

```text
D1902 = plcProductAteq1 或频道号
D1903 = plcProductAteq2 或频道号
```

### 6.5 启动检漏

进入 `StartJL` 后：

```text
确认 PLC 品番反馈
    -> requestLeakStart()
    -> D1900.5 脉冲
    -> 进入 JudgeResult
```

当前代码为了兼容现场，存在“跳过品番确认直接启动”的日志路径：

```text
[PLC] skip product channel confirm, start leak
```

实际启动仍通过 `D1900.5` 脉冲。

### 6.6 等待 PLC 总结果

`JudgeResult` 首先等待：

```text
D1901.2 = 1
```

然后读取：

```text
D1901.3
```

作为整件最终 OK/NG。

当前关键原则：

```text
PLC 总结果决定整件最终 OK/NG。
ATEQ 单路结果不覆盖 PLC 总结果。
```

### 6.7 ATEQ 单路显示

双路模式下：

- `D1901.6=1` 时，才根据 `D1901.7` 显示 ATEQ1 单路 OK/NG。
- `D1901.9=1` 时，才根据 `D1901.10` 显示 ATEQ2 单路 OK/NG。
- 某一路没有结果位时，不伪造 OK/NG。

ATEQ 串口返回后，主界面显示压力和泄漏值：

- ATEQ1 -> `led_Detection_Pressure`、`led_Leakage_value`
- ATEQ2 -> `led_Detection_Pressure_2`、`led_Leakage_value_2`

若某一路没有返回串口数据，则压力/泄漏值保持空。

### 6.8 结果收尾

当前结果收尾原则：

| 情况 | 行为 |
| --- | --- |
| PLC 总 NG | 允许进入收尾、排气、打印 NG、保存；不要求所有 ATEQ 数据齐 |
| PLC 总 OK | 仍等待本轮需要的 ATEQ 数据齐，或超时后走错误退出 |
| ATEQ 单路缺失 | 不伪造单路 NG，界面和数据库对应字段保持空 |

进入收尾函数：

```cpp
finishLeakCycleAfterResult()
```

其作用：

```text
v_Step_JL = Exhaust
v_Step_WorkModel = Reset
v_Step_JL_Reset = Exhaust
```

### 6.9 排气、打印、保存

`Step_JL_Reset()` 的 `Exhaust -> WaitExhaust` 阶段执行排气等待，之后判断结果。

当前 OK 条件：

```text
PLC 总结果不是 NG
并且 CurWorkPar.Result_Pisitive == OK
并且 CurWorkPar.Result_Negative == OK
```

当前 NG 条件：

```text
PLC 总 NG
或 CurWorkPar.Result_Pisitive == NG
或 CurWorkPar.Result_Negative == NG
```

NG 触发时：

```text
红灯
蜂鸣器
sign_Printe_Label_NG
SaveMysqlDatas
```

OK 触发时：

```text
绿灯
sign_Printe_Label_OK
SaveMysqlDatas
```

## 7. 当前结果判定细节

### 7.1 总结果

最终结果以 PLC 总结果为准：

```text
D1901.2 = 1 表示 PLC 总结果有效
D1901.3 = 1 表示 OK
D1901.3 = 0 表示 NG
```

### 7.2 单路结果

单路结果只用于显示，不决定整件最终结果：

```text
D1901.6/D1901.7 -> ATEQ1 单路显示
D1901.9/D1901.10 -> ATEQ2 单路显示
```

### 7.3 缺失数据处理

当前原则：

```text
没有单路结果位，不写 OK，不写 NG。
没有 ATEQ 串口返回，不写压力，不写泄漏值。
```

这避免了过去出现的错误：

```text
ATEQ2 没有返回值，却被软件显示成 NG。
```

### 7.4 PLC 总 NG 但 ATEQ2 缺失

示例：

```text
ATEQ1 NG 且有值
ATEQ2 无返回
PLC 总结果 NG
```

当前期望表现：

```text
总结果：NG
ATEQ1：显示 NG、压力、泄漏值
ATEQ2：留空
NG 标签：打印
数据库：ATEQ2 字段为空
```

## 8. 复位逻辑

### 8.1 软件复位按钮

主界面点击复位按钮：

```text
MFormJLStation::on_BTN_Reset_clicked
    -> m_WorkTask->requestManualReset()
    -> applyManualResetUi()
```

工作线程主循环中取出请求：

```text
takeManualResetRequest()
    -> performManualReset()
```

`performManualReset()` 会：

- 清条码。
- 清当前流程状态。
- 清 ATEQ 接收状态。
- 复位扫码前预复位状态。
- 发 `D1900.4` 全部复位脉冲。
- 发 `sign_applyManualResetUi()`。

### 8.2 实体复位

实体复位来源：

```text
N10 != 0
或 D1901.11 == 1
```

软件检测到实体复位后：

- 如果处于扫码触发/扫码前预复位过渡阶段，只清 `D1901.B` 锁存，不打断扫码。
- 其他阶段允许触发软件复位。

清锁存：

```cpp
SetDevice("D1901.B", 0)
```

### 8.3 扫码前预复位

下一轮扫码开始前会先发：

```text
D1900.4 全部复位脉冲
等待 500ms
再触发扫码枪
```

目的：

- 避免检测完成后立刻清三色灯结果。
- 让上一轮 OK/NG 保持到下一轮开始。
- 下一轮开始时再清 PLC 上一轮状态。

## 12. 当前已知风险和注意事项

### 12.1 D1901.1 同时承担检测中和扫码触发语义

当前代码使用 `D1901.1` 上升沿触发扫码，但 PLC 定义更像“整体检测中”。这会造成语义不够干净。

建议后续 PLC 提供独立启动/扫码请求位，例如：

```text
PLC_StartScan_Request
```

软件用独立上升沿触发扫码。

### 12.2 ATEQ 串口结果与 PLC 总结果解耦后，需要现场确认时序

当前设计：

- PLC 总结果用于最终 OK/NG。
- ATEQ 串口用于压力/泄漏值显示保存。

如果 PLC 过早给 OK，但 ATEQ 串口数据尚未返回，软件会等待 ATEQ 数据，超时后走错误退出。这个策略适合“OK 必须有完整数据”的要求。

如果后续现场要求“PLC OK 即可保存，ATEQ 数据可为空”，需要单独调整 OK 收尾策略。

### 12.3 ATEQ 单路 UI 的 OK/NG 文本来自 PLC 单路位

压力和泄漏值来自 ATEQ 串口。

OK/NG 文本来自：

```text
D1901.7
D1901.10
```

如果出现“有压力/泄漏值但没有 OK/NG 文本”，应检查 PLC 单路结果位。

如果出现“有 PLC 单路结果但没有压力/泄漏值”，应检查 ATEQ 串口接收和解析。

### 12.4 打印依赖 BarTender ActiveX

若出现：

```text
QAxBase::dynamicCallHelper: Object is not initialized
```

通常说明 BarTender COM 对象、模板文件、打印机驱动或权限有问题，不一定是检测流程问题。

### 12.5 数据库 SQL 字符串拼接存在风险

当前插入 SQL 使用字符串拼接。若条码、产品名、操作员等字段中包含单引号，可能导致 SQL 失败。

后续建议改为 `QSqlQuery::prepare()` + `bindValue()`。

### 12.6 QSettings 编码和中文乱码

当前大量旧中文注释/日志显示乱码，说明历史文件编码不统一。

后续建议：

- 源码统一 UTF-8。
- INI 继续兼容 GB2312 或逐步迁移。
- UI 显示文案和日志文案拆分管理。

### 13.1 建立正式通信协议文档

```cpp
struct LeakCycleResult {
```

## 15. 核心变量和状态含义

当前工程里有一些变量承担了跨模块语义。维护时最怕变量名看起来简单，但实际已经被多个流程复用。下面按类别整理。

### 15.1 当前产品/频道相关变量

| 变量/控件 | 所在模块 | 含义 |
| --- | --- | --- |
| `CurWorkPar.channel` | 全局工作参数 | 当前软件选中的频道/品番号 |
| `CurWorkPar.Product_Number` | 全局工作参数 | 当前品番编号或产品号显示内容 |
| `ui->CBT_Channel` | `MFormJLStation` | 主界面当前频道 |
| `ui->Channel` | `MFormWorkStep` | 品番设置界面当前频道 |
| `ui->CBT_Channel` | `MForm_PrintSetting` | 打印设置界面当前频道 |
| `D1902` | PLC | 软件写给 PLC 的产品/频道字，通常对应 ATEQ1 或主路 |
| `D1903` | PLC | 软件写给 PLC 的产品/频道字，通常对应 ATEQ2 或辅路 |
| `D90/D91` | PLC | PLC 反馈的当前选择/确认值 |

频道切换时最重要的是顺序：

```text
用户选择频道
-> 主界面同步 WorkStep 频道
-> 主界面加载当前频道工作参数
-> 主界面同步 PrintSetting 频道并加载打印参数
-> 主界面刷新品番显示
-> 主界面写 D1902/D1903 给 PLC
```

如果这个链路中使用 `setCurrentText()` 时不阻断信号，就容易出现重复进入槽函数、重复加载、甚至崩溃。

### 15.2 检测周期相关变量

| 变量 | 所在模块 | 含义 |
| --- | --- | --- |
| `v_Step_JL` | `mWorkTask` | 当前检漏流程状态 |
| `b_gameover` | `mWorkTask` | 当前周期是否结束 |
| `cycleAteqEnable` | `mWorkTask` | 本轮检测锁定的 ATEQ 模式 |
| `B_ATEQ1_Finished` | `mWorkTask` | ATEQ1 本轮是否已收到有效串口数据 |
| `B_ATEQ2_Finished` | `mWorkTask` | ATEQ2 本轮是否已收到有效串口数据 |
| `JL_Data_1` | `mWorkTask`/全局结果 | ATEQ1 结果、压力、泄漏值数据 |
| `JL_Data_2` | `mWorkTask`/全局结果 | ATEQ2 结果、压力、泄漏值数据 |
| `plcLeakResultLatched` | `mWorkTask` | PLC 总结果是否已锁存 |
| `plcOverallResultOk` | `mWorkTask` | PLC 总结果是否 OK |

重点：`cycleAteqEnable` 比界面当前 ATEQ 模式更可靠。因为检测过程中用户可能切换频道或重新加载品番，若直接读界面当前模式，可能用新频道解释旧周期结果。

### 15.3 扫码过渡相关变量

| 变量 | 含义 |
| --- | --- |
| `pendingBarcode` | 已收到但等待后续处理的条码 |
| `currentBarcode` | 当前周期正在使用的条码 |
| `pendingScanAfterPreReset` | PLC 触发扫码后，先复位再扫码的等待标志 |
| `preScanResetWaitMs` | 扫码前预复位等待时间计数 |
| `preResetAction` | 当前预复位动作类型 |

扫码过渡阶段是实体复位误触发最容易发生的位置。原因是扫码前会主动打 `D1900.4`，PLC 可能短时间残留 `D1901.11`，如果软件盲目把它当成人工复位，就会把刚开始的检测周期清掉。

### 15.4 复位相关变量

| 变量/信号 | 含义 |
| --- | --- |
| `lastPlcPhysicalReset` | 上一次是否检测到 PLC 实体复位请求，用于上升沿判断 |
| `physicalResetLatchHandled` | 当前 `D1901.11` 锁存是否已经处理过 |
| `D1901.11` / `D1901.B` | PLC 给软件的实体复位请求位 |
| `D1900.4` | 软件给 PLC 的全部复位/确认复位脉冲 |
| `requestManualReset()` | 外部请求软件复位 |
| `performManualReset()` | 实际执行软件复位流程 |

复位逻辑需要区分三类：

```text
软件复位按钮：用户在软件界面主动复位。
实体复位旋钮：PLC 检测实体按钮后，置 D1901.11 给软件。
扫码前预复位：下一件开始前，软件主动清上一轮 PLC 状态。
```

这三类复位不应该混成一个不分场景的“只要看到复位位就全部清空”。否则会出现启动/扫码刚开始就被清掉的问题。

## 16. 关键流程的细化时序

### 16.1 软件启动初始化时序

软件启动后大致按以下顺序建立运行环境：

```text
main
-> 创建主窗口 MFormJLStation
-> 创建/初始化子窗口和全局对象
-> 加载 INI 配置
-> 初始化扫码枪、ATEQ、PLC、数据库、打印设置
-> 建立信号槽连接
-> 启动工作线程或定时轮询
-> 主界面进入等待状态
```

启动阶段的风险点：

- 子窗口对象还没创建，主界面就访问其 `ui`。
- 下拉框加载默认值时自动触发 `currentTextChanged()`。
- 打印设置读取打印机列表时触发保存槽。
- PLC 未连接时仍执行写寄存器。
- BarTender ActiveX 未初始化成功，但后续仍调用 `dynamicCall()`。

因此在初始化阶段，所有跨窗口调用都应该有空指针保护；所有自动触发的 `setCurrentText()` 都要考虑是否需要 `QSignalBlocker`。

### 16.2 频道切换详细时序

频道切换当前应该保持下面的单向控制：

```text
主界面频道变化
-> 阻断 WorkStep 的 Channel 信号
-> 设置 WorkStep 当前频道
-> 主界面主动加载该频道 CurWorkPar
-> 阻断 PrintSetting 的 CBT_Channel 信号
-> 设置 PrintSetting 当前频道
-> 显式调用 PrintSetting 加载该频道打印参数
-> 刷新主界面 LE_Product_Number 等显示
-> 写 PLC 产品字 D1902/D1903
```

不能出现下面这种循环：

```text
主界面 set WorkStep Channel
-> WorkStep currentTextChanged
-> WorkStep set PrintSetting Channel
-> PrintSetting currentTextChanged
-> 又触发保存/加载
-> WorkStep Use 又 set 主界面 Channel
-> 主界面再次进入 currentTextChanged
```

如果后续再次出现“切换几次频道就崩溃”，应优先检查：

1. 是否新增了某个 `setCurrentText()` 没有加 `QSignalBlocker`。
2. 是否在 `currentTextChanged()` 里又调用了另一个控件的 `setCurrentText()`。
3. 是否某个槽函数里直接访问了尚未初始化的全局窗口对象。
4. 是否 `QSettings` 使用 `new` 后没有正确释放，或者错误调用了 `destroyed()`。
5. 是否 `invokeMethod()` 的槽名、参数类型和实际函数不匹配。

### 16.3 扫码枪直扫时序

扫码枪直扫不是 PLC 先触发扫码，而是扫码枪直接把条码发给软件。当前逻辑可理解为：

```text
扫码枪收到条码
-> MForm_ScanCode 发出条码信号
-> 主界面/工作线程接收条码
-> 软件先执行扫码前预复位 D1900.4
-> 等待预复位完成
-> 条码进入 mWorkTask
-> 校验条码规则
-> 写 D1902/D1903
-> 请求 PLC 启动检漏 D1900.5
```

这里的关键点是：直扫也需要预复位。原因是 ATEQ1/ATEQ2、PLC 结果位、三色灯等上一轮状态可能仍保持，如果不先清状态，新一轮可能会误读旧结果。

### 16.4 PLC 触发扫码时序

PLC 触发扫码通常来自 `D1901.1` 的上升沿。当前设计思想是：

```text
PLC 置扫码触发位
-> 软件检测上升沿
-> 软件先发 D1900.4 复位上一轮状态
-> 等待 500ms
-> 软件再触发扫码枪
-> 扫码枪返回条码
-> 后续同直扫流程
```

这样做的目的：

- 不在上一轮 OK/NG 结果刚出现时清三色灯。
- 让上一轮结果保持到下一轮真正开始。
- 下一轮开始前再清 PLC 状态，避免污染新周期。

### 16.5 PLC 启动检漏时序

软件启动检漏不是直接判定结果，而是给 PLC 一个启动脉冲：

```text
条码合格
-> 软件写 D1902/D1903 产品字
-> 软件确认或等待 PLC 产品反馈
-> 软件写 D1900.5 = 1
-> PLC 开始执行检漏动作
-> PLC 控制 ATEQ/工装/气路
-> PLC 最终给出 D1901.2/D1901.3 总结果
```

注意：PLC 是现场动作的核心控制者。软件不应该绕过 PLC 直接凭 ATEQ 串口值结束检测，因为 PLC 可能还在执行排气、夹具、节拍或安全互锁。

### 16.6 ATEQ 数据返回时序

ATEQ 串口数据通常形如：

```text
<01>:0.300 MPa:(OK):0.008 ml/min
```

软件解析后主要得到：

- 压力值：`0.300 MPa`
- 泄漏值：`0.008 ml/min`
- ATEQ 字符串中的 OK/NG 标记

但当前工程更稳妥的原则是：

```text
ATEQ 串口值用于显示和保存。
最终产品 OK/NG 以 PLC 总结果为准。
ATEQ 字符串的 OK/NG 不单独推翻 PLC 总结果。
```

原因：

- PLC 才知道整机动作是否完整。
- PLC 可能综合两个 ATEQ、气路动作、夹具状态、安全条件后给最终结论。
- 单独 ATEQ 返回 OK，不代表整件一定 OK。
- 单独 ATEQ 未返回，也不应该自动伪造另一路 NG 显示。

### 16.7 结果收尾时序

结果收尾当前应符合：

```text
PLC D1901.2 = 1
-> 软件读取 D1901.3
-> D1901.3 = 1：整件 OK
-> D1901.3 = 0：整件 NG
-> 软件更新结果灯
-> 软件等待或记录 ATEQ 已有数据
-> 软件进入排气/收尾
-> 打印 OK 或 NG 标签
-> 保存数据库
-> 本轮结束
```

关键边界：

- PLC 总 OK：软件最终应显示 OK，不能因为某一路 ATEQ 暂时无数据显示为 NG。
- PLC 总 NG：软件最终应显示 NG，并进入 NG 打印/保存，即使某一路 ATEQ 没有泄漏值。
- ATEQ 单路有值：显示对应压力/泄漏值。
- ATEQ 单路无值：保持空值或未获取状态，不伪造数值。

## 17. 结果判定的维护规则

这一节非常重要，后续改 OK/NG 相关代码前应该先看这里。

### 17.1 总原则

当前项目应遵守以下结果优先级：

```text
PLC 总结果 > 软件收尾流程 > ATEQ 单路显示
```

更明确地说：

```text
D1901.2 决定“PLC 是否已经有最终结果”。
D1901.3 决定“PLC 最终结果是 OK 还是 NG”。
D1901.6/D1901.7 可用于 ATEQ1 单路状态显示。
D1901.9/D1901.10 可用于 ATEQ2 单路状态显示。
ATEQ 串口字符串提供压力/泄漏值。
```

### 17.2 不应该再出现的错误逻辑

下面几类逻辑容易造成之前遇到的问题，后续应避免：

```text
错误 1：ATEQ2 没有返回，就把 ATEQ2 结果强行写成 NG。
后果：PLC 总 OK 时，软件却显示 NG，并且 ATEQ2 无压力/泄漏值。

错误 2：必须等 ATEQ1 和 ATEQ2 都有结果才处理 PLC 总 NG。
后果：PLC 已经报 NG，但因为某一路 ATEQ 没返回，软件不打印 NG 标签。

错误 3：检测过程中读取界面当前 ATEQ 模式，而不是本轮锁定模式。
后果：切频道或 Use 后，用新配置解释旧周期结果。

错误 4：把 ATEQ 串口字符串中的 OK/NG 当成整件最终结果。
后果：与 PLC 综合判定冲突。
```

### 17.3 PLC OK 但软件显示 NG 的排查顺序

如果现场再次出现“PLC 明明给 OK，软件却显示 NG”，建议按顺序排查：

1. 看日志中 `D1901` 的值，确认 `D1901.2` 和 `D1901.3` 是否同时符合总 OK。
2. 看 `cycleAteqEnable` 是否是本轮正确模式。
3. 看 `JL_Data_1` / `JL_Data_2` 的结果字段是否被某个超时逻辑提前写成 NG。
4. 看 `Step_JL_Reset()` 是否用“结果字符串非 OK”作为 NG 条件。
5. 看 ATEQ2 未返回时，代码是否仍补写了 `JL_Data_2.result = "NG"`。
6. 看 UI 显示是否只是单路 ATEQ 灯为 NG，而总结果灯其实是 OK。

正确逻辑应该是：

```text
PLC 总 OK：
    总结果 = OK
    有 ATEQ1 值就显示 ATEQ1 压力/泄漏值
    有 ATEQ2 值就显示 ATEQ2 压力/泄漏值
    没有某一路值，不自动把该路显示成 NG
```

### 17.4 PLC NG 但不打印 NG 标签的排查顺序

如果出现“PLC 报 NG，但软件不打印 NG 标签”，建议查：

1. `D1901.2` 是否为 1。如果 PLC 只给了单路 NG，但没有给总结果有效位，软件可能不知道最终完成。
2. `D1901.3` 是否为 0。如果总结果位语义反了，需要和 PLC 重新确认。
3. `plcLeakResultLatched` 是否被置位。
4. `plcOverallResultOk` 是否为 false。
5. `Step_JL_Reset()` 是否因为等待 ATEQ 双路结果而没有进入收尾。
6. `slot_Printe_Label_NG()` 是否被调用。
7. NG 模板路径 `/label/NG.btw` 是否存在。
8. BarTender ActiveX 是否初始化成功。

正确逻辑应该是：

```text
PLC 总 NG：
    总结果 = NG
    允许已有 ATEQ 数据显示
    缺失 ATEQ 数据不阻止 NG 收尾
    进入 NG 打印
    进入数据库保存
```

### 17.5 双 ATEQ 模式的建议解释方式

双 ATEQ 模式下，建议把结果分成三层：

```text
整件结果：
    来自 PLC 总结果 D1901.2/D1901.3。

ATEQ1 单路状态：
    来自 PLC 单路位或 ATEQ1 实际返回，仅用于显示和追溯。

ATEQ2 单路状态：
    来自 PLC 单路位或 ATEQ2 实际返回，仅用于显示和追溯。
```

数据库也应该尽量保留这种结构：

| 字段 | 建议含义 |
| --- | --- |
| `testing_result` | 整件最终结果，来自 PLC 总结果 |
| `testing_pressuse` | ATEQ1 压力 |
| `testing_leackage_value` | ATEQ1 泄漏值 |
| `testing_result_2` | ATEQ2 单路状态 |
| `testing_pressuse_2` | ATEQ2 压力 |
| `testing_leackage_value_2` | ATEQ2 泄漏值 |

如果 PLC 总 NG 但 ATEQ2 无返回，推荐保存：

```text
testing_result = NG
testing_result_2 = 空或未获取
testing_pressuse_2 = 空
testing_leackage_value_2 = 空
```

不要为了让 NG 打印触发而把 ATEQ2 伪造成 NG。NG 打印应该由整件结果触发，而不是由 ATEQ2 字段触发。

## 18. 复位逻辑的边界和排查

### 18.1 三种复位的区别

| 类型 | 触发方 | 软件动作 | PLC 动作 | 典型用途 |
| --- | --- | --- | --- | --- |
| 软件复位按钮 | 人在软件界面点击 | 清软件状态、UI、ATEQ 接收状态，并发 D1900.4 | PLC 收到 D1900.4 后清现场状态 | 手动恢复 |
| 实体复位旋钮 | PLC 置 D1901.11 | 软件判断是否是真正人工复位，再执行软件复位 | PLC 负责检测旋钮并置位/清位 | 现场人员操作 |
| 扫码前预复位 | 软件在下一轮开始前触发 | 只为新一轮清上一轮 PLC 状态 | PLC 清上一轮结果/灯/状态 | 防止旧状态污染新周期 |

维护时要记住：

```text
不是所有 D1900.4 都代表人工复位。
不是所有 D1901.11 都应该立即清空软件流程。
```

### 18.2 实体复位为什么不能盲目触发

实体复位曾经出现的问题是：启动或扫码刚发生，软件立即检测到 `D1901.11 = 1`，于是执行软件复位，把刚进入的检测流程清掉。

这通常不是现场真的按了复位，而可能是：

- PLC 锁存位没有及时清零。
- 软件扫码前预复位导致 PLC 短时置位。
- 上一轮残留状态在下一轮开始时被读到。
- PLC 侧把某个复位确认/清状态动作映射到了 `D1901.11`。

所以当前更合理的规则是：

```text
启动/扫码过渡瞬间：
    读到 D1901.11，只清 D1901.B，不执行软件复位。

检测中、NG 后、异常中、空闲时：
    读到 D1901.11，可以执行软件复位。
```

### 18.3 NG 后为什么必须允许实体复位

NG 后现场通常需要人工确认、处理工件、旋钮复位，然后软件也应恢复到初始状态。如果代码简单写成“检测中或非空闲不允许复位”，就会导致 NG 后复位不了，现场进入死锁。

因此判断重点不是“是否检测中”，而是“是否处在扫码/启动的短暂过渡误触发窗口”。

建议判断思路：

```text
if D1901.11 == 1:
    if 正在扫码前预复位 or 条码刚进入但尚未稳定启动:
        清 D1901.B
        不执行软件复位
    else:
        执行 performManualReset()
        清 D1901.B
```

### 18.4 复位相关日志怎么看

重点日志包括：

```text
[PLC] physical reset trigger
[PLC] physical reset ignored, scan transition
[PLC] manual reset requested
[PLC] request all reset pulse D1900 bit4
[PLC] clear physical reset latch D1901.B
```

如果按实体复位没有效果：

1. 看是否有 `D1901=2048` 或包含 bit11 的值。
2. 看是否打印 `physical reset trigger`。
3. 如果只打印 `ignored`，说明软件认为处于扫码过渡阶段。
4. 看是否调用了 `manual reset requested`。
5. 看 UI 是否收到复位信号。
6. 看 `D1901.B` 是否清零成功。

如果启动后立即被复位：

1. 看启动/扫码时间点附近是否有 `physical reset trigger`。
2. 如果有，说明误把残留 `D1901.11` 当成人工复位。
3. 需要确认扫码过渡屏蔽条件是否覆盖该时刻。

## 19. 打印和数据库的细化说明

### 19.1 OK 标签打印条件

OK 标签应由整件最终 OK 触发。典型路径：

```text
PLC 总结果 OK
-> 工作线程进入 OK 收尾
-> 主界面或工作线程发出 OK 打印信号
-> MForm_PrintSetting::slot_Printe_Label_OK(channel)
-> 加载当前频道打印参数
-> 流水号 +1
-> 写 channel_x.txt
-> 打开 channel_x.btw
-> BarTender 打印
```

OK 打印涉及：

- 当前频道 `channel`
- 当前标签流水号 `LE_Number`
- 打印模板 `/label/channel_%1.btw`
- 文本文件 `/label/channel_%1.txt`
- OK 打印机 `printer_driver_ok`

### 19.2 NG 标签打印条件

NG 标签应由整件最终 NG 触发。典型路径：

```text
PLC 总结果 NG
-> 工作线程进入 NG 收尾
-> 发出 NG 打印信号
-> MForm_PrintSetting::slot_Printe_Label_NG(channel, leak_value)
-> 加载当前频道打印参数
-> 写 NG.txt
-> 打开 NG.btw
-> BarTender 打印
```

NG 打印不应该依赖 ATEQ2 是否有值。否则会出现 PLC 已经 NG，但因为 ATEQ2 没有泄漏值而不打印 NG 标签的问题。

### 19.3 BarTender ActiveX 风险

日志中曾出现：

```text
QAxBase::dynamicCallHelper: Object is not initialized, or initialization failed
```

这个警告通常说明：

- BarTender 未安装。
- ActiveX 控件注册失败。
- `m_btApp->setControl("BarTender.Application")` 失败。
- `querySubObject("Formats")` 返回空。
- 模板路径不存在，导致 `Open()` 失败。
- 打印机名称为空或打印机不可用。

建议后续增强 `slot_printer()`：

```text
检查 m_btApp 是否为空
检查 m_btApp->isNull()
检查模板文件是否存在
检查 printer_name 是否为空
检查 Formats 是否为空
检查 Open() 返回 m_format 是否为空
检查 PrintSetup 是否为空
失败时输出明确日志并 return
```

### 19.4 数据库存储字段含义

当前插入 `jl_datas` 的字段大致包括：

| 字段 | 含义 |
| --- | --- |
| `product_number` | 品番或产品编号 |
| `eq_number` | 设备编号 |
| `channel` | 当前频道 |
| `product_code` | 条码或产品码 |
| `batch` | 批次 |
| `operator_id` | 操作员 |
| `testing_start_time` | 检测开始时间 |
| `testing_finished_time` | 检测结束时间 |
| `testing_time` | 检测用时 |
| `testing_result` | 整件最终结果 |
| `testing_pressuse` | ATEQ1 压力 |
| `testing_leackage_value` | ATEQ1 泄漏值 |
| `testing_result_2` | ATEQ2 单路结果 |
| `testing_pressuse_2` | ATEQ2 压力 |
| `testing_leackage_value_2` | ATEQ2 泄漏值 |
| `finished_product_code` | 打印生成的成品码 |

后续建议把数据库保存也按“整件结果”和“单路数据”拆清楚，避免用 ATEQ 单路字段倒推整件结果。

## 20. 现场问题排查手册

### 20.1 切频道崩溃

优先看日志最后停在哪里：

```text
[CHANNEL] channel change begin
[CHANNEL] workstep channel synced
[CHANNEL] work params loaded
[CHANNEL] print channel synced
[CHANNEL] product text synced
[CHANNEL] product words written
```

定位方法：

| 最后一条日志 | 可能问题 |
| --- | --- |
| 只到 `begin` | WorkStep 对象或 UI 空指针 |
| 到 `workstep channel synced` | 加载频道参数时崩溃 |
| 到 `work params loaded` | 打印设置对象或 UI 空指针 |
| 到 `print channel synced` | 打印参数加载或 QSettings 问题 |
| 到 `product text synced` | PLC 写产品字崩溃或通信对象异常 |

### 20.2 扫码后没有启动检漏

排查顺序：

1. 扫码枪是否输出条码日志。
2. 条码是否通过长度/规则校验。
3. `CurWorkPar.channel` 是否正确。
4. `D1902/D1903` 是否写入成功。
5. `D90/D91` 是否反馈匹配。
6. 是否发出 `D1900.5` 启动脉冲。
7. PLC 是否收到启动脉冲。
8. 是否被实体复位误触发清掉流程。

### 20.3 ATEQ 有返回但界面无压力/泄漏值

排查顺序：

1. ATEQ 串口日志是否有完整字符串。
2. 字符串是否被解析成压力和泄漏值。
3. 当前周期是否仍处于允许接收 ATEQ 数据的状态。
4. `activeAteqEnable()` 是否包含该路。
5. ATEQ1/ATEQ2 是否接反。
6. UI 更新信号是否连接到 `slot_updateAteqDisplay()`。
7. 是否在接收前已经被复位或收尾清空。

### 20.4 PLC OK 但软件 NG

重点查：

```text
D1901.2 是否为 1
D1901.3 是否为 1
JL_Data_1/JL_Data_2 是否被提前写 NG
ATEQ2 缺失是否被补 NG
Step_JL_Reset 是否用单路字段判整件 NG
```

正确方向：

```text
PLC 总 OK 就应整件 OK。
ATEQ 单路无值只影响该路数据显示，不应该推翻总 OK。
```

### 20.5 PLC NG 但软件不打印 NG

重点查：

```text
D1901.2 是否为 1
D1901.3 是否为 0
plcLeakResultLatched 是否为 true
plcOverallResultOk 是否为 false
是否进入 NG 收尾
是否调用 slot_Printe_Label_NG
NG.btw 是否存在
BarTender 是否可用
```

正确方向：

```text
PLC 总 NG 应独立触发 NG 收尾。
不能因为 ATEQ2 无值卡住 NG 打印。
```

### 20.6 实体复位无效

重点查：

```text
D1901 是否包含 2048
日志是否出现 physical reset trigger
是否被判断为 scan transition ignored
是否调用 manual reset requested
是否发出 UI 复位信号
D1901.B 清零是否成功
```

如果 `D1901.11` 一直为 1：

- 软件上升沿逻辑只会触发一次。
- 后续要再次触发，需要 PLC 先清 0 再置 1。
- 如果 PLC 不清零，软件可以写 `D1901.B = 0` 尝试清锁存，但最终仍要确认 PLC 侧是否允许软件清。

### 20.7 打印失败

排查顺序：

1. 打印机名称是否加载正确。
2. `NG.btw` 或 `channel_x.btw` 是否存在。
3. `NG.txt` 或 `channel_x.txt` 是否写入成功。
4. BarTender 是否安装。
5. ActiveX 是否注册。
6. `m_btApp` 是否初始化成功。
7. 打印机是否在线。
8. 是否有权限访问模板路径。

## 21. 后续修改代码时的边界清单

### 21.1 修改结果逻辑时不能破坏的边界

必须保持：

```text
整件最终结果以 PLC 总结果为准。
ATEQ 单路值用于显示和追溯。
缺失 ATEQ 单路不自动伪造成 NG。
PLC 总 NG 可以直接触发 NG 打印。
PLC 总 OK 不应被 ATEQ 缺值改成 NG。
```

如果业务后来明确要求“ATEQ 缺值也算 NG”，应新增独立故障类型，例如：

```text
Result = NG
Reason = ATEQ2_TIMEOUT
```

而不是把 `ATEQ2` 的结果字段直接伪造成普通 NG。这样后续追溯才能分清“真实泄漏 NG”和“通信缺失 NG”。

### 21.2 修改复位逻辑时不能破坏的边界

必须保持：

```text
软件复位按钮永远可以复位。
NG 后实体复位可以复位。
异常中实体复位可以复位。
扫码/启动短暂过渡期不能被残留 D1901.11 清掉。
扫码前预复位不能被误认为人工复位。
```

建议后续给复位增加原因枚举：

```cpp
enum class ResetReason {
    SoftwareButton,
    PhysicalButton,
    PreScan,
    PlcError,
    Emergency
};
```

这样日志和逻辑都会更清楚。

### 21.3 修改频道逻辑时不能破坏的边界

必须保持：

```text
主界面是频道切换的统一入口。
WorkStep 的 Use 可以请求主界面切换，但不应形成循环触发。
PrintSetting 需要随频道加载一次，但不能被重复触发。
写 D1902/D1903 应在 CurWorkPar 更新后进行。
```

新增任何频道同步代码前，都应该检查：

- 是否会触发 `currentTextChanged()`。
- 是否需要 `QSignalBlocker`。
- 是否已有显式加载函数。
- 是否可能访问空 `ui`。

### 21.4 修改打印逻辑时不能破坏的边界

必须保持：

```text
OK 打印只在整件 OK 后触发。
NG 打印只在整件 NG 后触发。
打印失败不应导致程序崩溃。
模板不存在只输出日志，不直接访问空 ActiveX 对象。
```

后续如果要让打印失败阻止下一件，应当作为新业务规则明确实现，而不是靠崩溃或异常副作用阻断流程。

### 21.5 修改 PLC 通信时不能破坏的边界

必须保持：

```text
D1900 是软件写 PLC 的控制字。
D1901 是 PLC 给软件的状态/结果字，除明确允许的 D1901.B 清锁存外，不随意写。
D1902/D1903 是产品/频道字。
D90/D91 是 PLC 反馈/确认字。
```

所有位定义都应和 PLC 工程同步确认。如果 PLC 侧改了位语义，软件文档和代码注释也要同步更新。

### 21.6 推荐新增的工程内文档

当前这份文档是总览。后续建议继续拆成几份更专门的文档：

```text
docs/PLC通信协议.md
docs/检漏状态机.md
docs/ATEQ数据解析说明.md
docs/打印与数据库字段说明.md
docs/现场故障排查手册.md
docs/版本变更记录.md
```

这样后续每次修改都能明确更新对应文档，不会把所有细节堆在一份大文档里。

## 22. 总结

当前工程已经形成了一个可运行的完整检漏工站闭环：

```text
扫码
-> 条码校验
-> PLC 品番写入
-> PLC 启动检漏
-> PLC 总结果判定
-> ATEQ 压力/泄漏值显示
-> 排气收尾
-> OK/NG 标签打印
-> 数据库存储
-> 等待下一轮
```

当前最重要的逻辑边界是：

```text
PLC 总结果决定整件最终 OK/NG。
ATEQ 单路只负责对应路显示和数据。
缺失的 ATEQ 单路不伪造成 NG。
PLC 总 NG 可以独立触发 NG 标签。
```

这条边界是后续维护时必须保持的核心规则。只要继续遵守这个规则，就能避免之前出现的“ATEQ2 无值却显示 NG”“PLC 已报 NG 但不打印标签”等问题。

后续提升重点建议集中在三方面：

1. 协议文档化：把 PLC 位定义、清零方、触发方式写成正式版本。
2. 状态机清晰化：把 `Step_JL()` 和 `Step_JL_Reset()` 的历史逻辑拆成明确状态。
3. 数据模型结构化：用独立结构保存 PLC 总结果、ATEQ 单路结果、ATEQ 串口数据，避免字段复用造成逻辑混乱。
