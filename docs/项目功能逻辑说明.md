# Qt_SH_JLStation 项目功能逻辑说明

更新时间：2026-06-01

## 1. 文档目的

本文档用于说明当前项目中扫码、PLC、ATEQ 检漏、复位、结果保存和打印之间的实际代码逻辑。

当前项目是一个改造项目，原框架仍保留较多历史变量和旧流程代码。本说明按“当前实际执行链路”为主，同时标出仍存在风险或需要 PLC 侧确认的点，避免把未闭环内容误认为已经稳定。

## 2. 当前核心闭环

当前程序目标流程为：

```text
PLC 状态触发扫码
    -> 扫码枪接收条码
    -> 主界面回填条码输入框
    -> 工作线程接收条码
    -> 条码规则校验
    -> 写 D1902/D1903 品番与 D1900 ATEQ 使能
    -> 软件通过 D1900.5 发启动检漏脉冲
    -> 等 PLC D1901 检漏结果位
    -> 等 ATEQ 串口返回压力/泄漏值
    -> 软件通过 D1900.4 发全部复位脉冲
    -> 进入原复位收尾
    -> 保存/打印
    -> 清本轮状态
    -> 回到空闲，等待下一轮
```

当前界面状态：

| 控件 | 当前状态 | 说明 |
| --- | --- | --- |
| 开始按钮 | 隐藏 | 槽函数保留，但正常流程不依赖它 |
| 复位按钮 | 显示 | 最终测试可手动点击，执行整轮软件复位 |
| 急停显示 | 隐藏 | 界面不显示，急停保护逻辑仍保留 |

## 3. 主要模块职责

### 3.1 MFormJLStation 主界面

文件：

- `MFormJLStation.cpp`
- `MFormJLStation.h`

主要职责：

- 初始化主界面、扫码枪、PLC、ATEQ、数据库、打印模块。
- 接收扫码枪条码信号。
- 回填条码到 `LE_ProductCode`。
- 将条码和界面快照交给工作线程。
- 接收 ATEQ 显示信号，更新压力和泄漏值。
- 点击复位按钮时调用工作线程人工复位接口。

关键函数：

| 函数 | 作用 |
| --- | --- |
| `slot_onBarcodeReceived()` | 接收扫码枪条码，回填输入框，交给工作线程 |
| `slot_updateAteqDisplay()` | 接收 ATEQ 串口解析结果，显示压力/泄漏值，并通知工作线程 |
| `on_BTN_Reset_clicked()` | 人工复位按钮，调用 `m_WorkTask->requestManualReset()` |
| `triggerScanCode()` | 发送扫码枪触发命令 |
| `slot_init_Display()` | 清界面显示 |

### 3.2 mWorkTask 工作线程

文件：

- `mWorkTask.cpp`
- `mWorkTask.h`

主要职责：

- 管理整轮测试状态机。
- 接收并缓存条码。
- 校验条码规则。
- 写入品番。
- 发起软件启动检漏。
- 等待 PLC 检漏结果。
- 等待 ATEQ 串口数据。
- 触发全部复位。
- 进入原有复位收尾、保存、打印流程。

关键函数：

| 函数 | 作用 |
| --- | --- |
| `WorkTask()` | 工作线程主循环 |
| `acceptBarcode()` | 线程安全接收主界面确认后的条码 |
| `takePendingBarcode()` | 工作线程消费待处理条码 |
| `Step_JL()` | 正常检测状态机 |
| `resetLeakCycleState()` | 初始化本轮 ATEQ/PLC 结果状态 |
| `acceptAteqData()` | 线程安全接收 ATEQ 串口结果 |
| `requiredAteqDataReceived()` | 判断当前 ATEQ 模式所需串口数据是否齐全 |
| `requestReceivedLeakResets()` | ATEQ 数据齐全后触发 D1900.4 全部复位 |
| `finishLeakCycleAfterResult()` | 检漏结果完成后进入复位收尾 |
| `requestManualReset()` | UI 线程请求人工复位，只置位请求标志 |
| `performManualReset()` | 工作线程内部执行人工复位，避免跨线程直接改状态 |

### 3.3 mModbusRtu PLC 通信

文件：

- `ModbusRtu/mModbusRtu.cpp`
- `ModbusRtu/mModbusRtu.h`

主要职责：

- 读 PLC 输入寄存器。
- 写 PLC 输出寄存器。
- 维护 D1900/D1901/D1902/D1903/D90/D91 镜像。
- 根据当前 ATEQ 选择生成使能、品番、复位、启动输出。

关键函数：

| 函数 | 作用 |
| --- | --- |
| `syncOutputsToPlcWords()` | 生成并写入 D1900/D1902/D1903 |
| `syncPlcWordsToInputs()` | 把 D1901 映射回原框架输入状态 |
| `writeCurrentProductWords()` | 条码通过后立即写 D1902/D1903 |
| `requestLeakStart()` | 请求 D1900.5 启动检漏脉冲 |
| `requestAllReset()` | 请求 D1900.4 全部复位脉冲 |
| `plcSelectedLeakHasResult()` | 判断当前 ATEQ 选择对应 PLC 结果是否到达 |
| `plcSelectedLeakIdle()` | 判断当前 ATEQ 选择对应检漏是否回到空闲 |

### 3.4 MForm_ScanCode 扫码枪

文件：

- `Honeywell/MForm_ScanCode.cpp`
- `Honeywell/MForm_ScanCode.h`

主要职责：

- 打开扫码枪串口。
- 写入扫码触发命令。
- 聚合串口数据。
- 收到完整条码后发出 `sign_barcodeReceived(QString)`。

关键点：

- 扫码枪触发命令为：

```text
0x16 0x54 0x0D
```

- 串口未打开、不可写时不会直接崩溃，而是发 `sign_scanWriteFailed()`。
- 条码读取后执行 `trimmed()`，非空才发给主界面。

### 3.5 ATEQ 串口

文件：

- `MATEQ/combase.cpp`

主要职责：

- ATEQ1 串口接收。
- ATEQ2 串口接收。
- 解析 ATEQ 返回字符串。
- 发送 `sign_ATEQ_Display(device, pressure, leakage)` 到主界面。

当前固定映射：

| 设备 | 检漏仪 | 串口 | PLC 品番 | 使能位 | PLC 结果位 |
| --- | --- | --- | --- | --- | --- |
| ATEQ1 | 检漏仪1 | COM4 | D1902 | D1900.3 | D1901.6/7 |
| ATEQ2 | 检漏仪2 | COM2 | D1903 | D1900.9 | D1901.9/10 |

ATEQ 字符解析规则：

示例：

```text
检漏1:<01>:0.300 MPa:(OK):0.073 ml/min
```

解析方式：

- 以冒号 `:` 拆分。
- 查找包含 `OK` 或 `NG` 的片段。
- 结果片段前一段为压力。
- 结果片段后一段为泄漏值。

解析结果：

| 字段 | 示例 |
| --- | --- |
| pressure | `0.300 MPa` |
| result | `OK` |
| leakage | `0.073 ml/min` |

## 4. PLC 点位说明

### 4.1 软件写 PLC

| 寄存器 | Bit | 当前定义 | 当前代码行为 |
| --- | --- | --- | --- |
| D1900 | 0 | 心跳 | 周期翻转 |
| D1900 | 1 | 软件准备完毕 | 固定置 1 |
| D1900 | 3 | 检漏1使能 | ATEQ1 或双检时置 1 |
| D1900 | 4 | 全部复位 | 复位脉冲，约 300ms |
| D1900 | 5 | 软件启动检漏 | 启动检漏脉冲，约 300ms |
| D1900 | 9 | 检漏2使能 | ATEQ2 或双检时置 1 |
| D1900 | 10 | 停用 | 固定清 0 |
| D1902 | - | 检漏1目标品番/频道 | ATEQ1 或双检时写当前软件频道号 |
| D1903 | - | 检漏2目标品番/频道 | ATEQ2 或双检时写当前软件频道号 |

注意：

- 当前代码没有 `D1900.11`。
- 当前代码没有把 `D1900.10` 作为检漏2复位使用。
- 当前代码不写 `D90/D91/D1901`。

### 4.2 PLC 给软件

| 寄存器 | Bit | 定义 | 当前用途 |
| --- | --- | --- | --- |
| D1901 | 0 | 工装到位 | 原框架输入映射，当前备注为不使用 |
| D1901 | 1 | 整体检测状态 | 当前代码仍用其上升沿触发扫码，这是风险点 |
| D1901 | 2 | 整体检测已有结果 | 读入并记录 |
| D1901 | 3 | 整体检测结果 | 读入并记录 |
| D1901 | 5 | 检漏1检测中 | 用于判断空闲 |
| D1901 | 6 | 检漏1有结果 | ATEQ1 结果到达判断 |
| D1901 | 7 | 检漏1结果 | ATEQ1 OK/NG |
| D1901 | 8 | 检漏2检测中 | 用于判断空闲 |
| D1901 | 9 | 检漏2有结果 | ATEQ2 结果到达判断 |
| D1901 | 10 | 检漏2结果 | ATEQ2 OK/NG |
| D90 | - | PLC 当前检漏1编号 | 只读，用于确认检漏1已切到当前软件频道 |
| D91 | - | PLC 当前检漏2编号 | 只读，用于确认检漏2已切到当前软件频道 |

注意：

- 当前代码不再读取 `D9101`。
- 当前代码没有 `D1901.11`。
- 当前代码没有独立的“PLC 实体启动按钮请求扫码”输入位。

## 5. 详细流程解析

### 5.1 程序启动

主界面构造函数完成以下初始化：

1. 初始化 UI。
2. 加载 INI 配置。
3. 创建打印设置窗口。
4. 创建工作参数窗口。
5. 创建扫码枪窗口。
6. 创建工作线程 `mWorkTask`。
7. 建立扫码枪信号到主界面的连接。
8. 建立工作线程信号到主界面的连接。
9. 初始化 PLC ActiveX 控件。
10. 创建 PLC 配置窗口和 ATEQ 窗口。
11. 建立 ATEQ 串口结果到主界面的连接。
12. 启动主界面定时器。
13. 启动工作线程。

界面隐藏策略：

- 隐藏开始按钮。
- 显示复位按钮。
- 隐藏急停显示。
- 隐藏部分旧状态灯。

### 5.2 PLC 读写循环

PLC 通信模块周期读写：

读：

```text
D1901
D90
D91
```

写：

```text
D1900
D1902
D1903
```

写入时会根据当前参数生成：

```text
D1900.0 心跳
D1900.1 软件准备
D1900.3 检漏1使能
D1900.4 全部复位脉冲
D1900.5 启动检漏脉冲
D1900.9 检漏2使能
D1900.10 固定 0
D1902 检漏1品番
D1903 检漏2品番
```

### 5.3 扫码触发

当前代码在工作线程主循环中读取：

```cpp
const bool plcOverallTesting = m_ModbusRtu->plcOverallTesting();
```

`plcOverallTesting()` 实际读取：

```text
D1901 bit1
```

当检测到 `D1901.1` 从 0 变 1 时，当前代码执行扫码枪触发。

当前触发前会检查：

- 工位是否空闲。
- 是否已有条码。
- 是否急停。
- 当前通道是否合法。
- ATEQ 选择是否合法。

如果通过：

- 清界面。
- 发扫码枪触发命令。
- 检测时间开始计时。
- 扫码灯置黄。

重要风险：

`D1901.1` 按 PLC 表定义是“整体检测状态”，不是“实体启动按钮请求扫码”。因此这部分目前是借用状态位触发扫码，严格来说不够干净。若 PLC 后续提供单独的“启动按钮请求扫码”输入位，应该替换此逻辑。

### 5.4 条码接收

扫码枪串口收到数据后：

1. `slot_readRead()` 累积串口数据。
2. 50ms 定时结束后进入 `slot_readAll()`。
3. 将字节转为字符串。
4. 执行 `trimmed()`。
5. 非空则发出 `sign_barcodeReceived(barcode)`。

主界面收到条码后：

1. 清除前后空格。
2. 写入 `LE_ProductCode`。
3. 检查通道是否合法。
4. 检查工作线程是否存在。
5. 生成测试快照。
6. 调用 `m_WorkTask->acceptBarcode()`。

工作线程 `acceptBarcode()` 会检查：

- 条码非空。
- 工位空闲。
- 没有待处理条码。
- 没有当前条码。
- 没有急停。

通过后写入：

```text
pendingBarcode
pendingSnapshot
```

### 5.5 条码进入工作线程

工作线程主循环调用 `takePendingBarcode()` 消费条码：

1. `pendingBarcode` 转为 `currentBarcode`。
2. `pendingSnapshot` 转为 `currentSnapshot`。
3. 清空 pending。
4. 设置本轮状态：
   - `b_gameover = false`
   - `v_Step_JL = WaitGetCode`
   - `v_Step_WorkModel = Normal`
   - 清上一轮结果
   - 扫码灯黄
   - 检测时间黄

### 5.6 条码校验

在 `Step_JL()` 的 `WaitGetCode` 阶段：

1. 判断 `currentBarcode` 是否非空。
2. 回填条码输入框。
3. 停止扫码等待计时。
4. 清 ATEQ 数据缓存。
5. 判断上一件 NG 是否需要光栅放置。
6. 校验条码固定段。
7. 校验条码长度。

通过后：

- 检查 ATEQ 是否合法。
- 写入 PLC 品番。
- 扫码灯置绿。
- 结果灯置黄。
- 进入 `StartJL`。

失败后：

- 扫码灯置红。
- 本轮结束。
- 清当前条码。
- 不启动检漏。

### 5.7 品番和 ATEQ 使能写入

条码通过后调用：

```cpp
m_ModbusRtu->writeCurrentProductWords()
```

写入规则：

| ATEQ 模式 | D1902 | D1903 | D1900.3 | D1900.9 |
| --- | --- | --- | --- | --- |
| ATEQ1 | 写当前软件频道号 | 写当前软件频道号 | 1 | 0 |
| ATEQ2 | 写当前软件频道号 | 写当前软件频道号 | 0 | 1 |
| 双检 | 写当前软件频道号 | 写当前软件频道号 | 1 | 1 |

`D1900.3` 和 `D1900.9` 在周期写入 `syncOutputsToPlcWords()` 中持续按当前 ATEQ 模式刷新。

检漏仪1、检漏仪2的目标频道会一起更新；由 `D1900.3` 和 `D1900.9` 控制本轮实际启用哪台检漏仪。

当前规则固定为：软件频道号等于检漏仪 PR 号。旧的 `plc_product_ateq1/plc_product_ateq2` 字段不再参与 PLC 下发值。

启动检漏前会读取 PLC 当前编号并确认：

| ATEQ 模式 | 启动前确认 |
| --- | --- |
| ATEQ1 | `D90 == 当前软件频道号` |
| ATEQ2 | `D91 == 当前软件频道号` |
| 双检 | `D90 == 当前软件频道号 && D91 == 当前软件频道号` |

确认超时则不发 `D1900.5`。

### 5.8 软件启动检漏

条码通过后进入 `StartJL`。

`StartJL` 首次执行时：

```cpp
m_ModbusRtu->requestLeakStart();
```

该函数设置：

```text
leak_start_ticks = 3
```

PLC 写入周期中：

```text
D1900.5 = leak_start_ticks > 0 && all_reset_ticks == 0
```

含义：

- `D1900.5` 为软件启动检漏脉冲。
- 当前按 3 个写入周期保持，约 300ms。
- 如果正在复位脉冲中，则不允许启动检漏。

随后等待一小段时间，进入 `JudgeResult`。

### 5.9 PLC 结果等待

进入 `JudgeResult` 后，代码不会直接判定 OK/NG，而是调用：

```cpp
m_ModbusRtu->plcSelectedLeakHasResult()
```

判断规则：

| ATEQ 模式 | 等待条件 |
| --- | --- |
| ATEQ1 | D1901.6 = 1 |
| ATEQ2 | D1901.9 = 1 |
| 双检 | D1901.6 = 1 且 D1901.9 = 1 |

结果读取：

| ATEQ 模式 | 结果位 |
| --- | --- |
| ATEQ1 | D1901.7 |
| ATEQ2 | D1901.10 |
| 双检 | D1901.7 和 D1901.10 |

拿到 PLC 结果后：

- 停止检测时间。
- 更新总结果灯。
- 更新项目结果文字。
- 更新项目压力/泄漏值区域颜色。
- 锁存 `plcLeakResultLatched = true`。
- 开始等待 ATEQ 串口数据。

### 5.10 ATEQ 串口等待

PLC 结果到达后，代码仍需等待 ATEQ 串口数据。

等待规则：

| ATEQ 模式 | 需要收到的数据 |
| --- | --- |
| ATEQ1 | ATEQ1 串口数据 |
| ATEQ2 | ATEQ2 串口数据 |
| 双检 | ATEQ1 和 ATEQ2 都收到 |

ATEQ 数据由主界面 `slot_updateAteqDisplay()` 处理：

1. 判断设备是否属于当前 ATEQ 模式。
2. 调用 `m_WorkTask->acceptAteqData()`。
3. 更新压力和泄漏值显示。

工作线程中 `acceptAteqData()` 会通过互斥锁设置：

```text
ateq1DataReceived
ateq2DataReceived
```

如果 10 秒内未收到所需 ATEQ 数据：

- 输出 timeout 日志。
- 结果灯置红。
- 本轮结束。
- 清条码。

### 5.11 全部复位

ATEQ 所需数据全部收到后：

```cpp
requestReceivedLeakResets()
```

只触发一次：

```cpp
m_ModbusRtu->requestAllReset();
```

`requestAllReset()` 设置：

```text
all_reset_ticks = 3
leak_start_ticks = 0
```

PLC 写入周期中：

```text
D1900.4 = all_reset_ticks > 0
D1900.5 = leak_start_ticks > 0 && all_reset_ticks == 0
D1900.10 = 0
```

含义：

- D1900.4 是全部复位。
- 复位期间会强制取消启动检漏脉冲。
- D1900.10 永远不会作为复位输出置 1。

### 5.12 检测完成收尾

ATEQ 数据齐全并触发全部复位后：

```cpp
finishLeakCycleAfterResult()
```

该函数将状态切到：

```text
v_Step_JL = Exhaust
v_Step_WorkModel = Reset
v_Step_JL_Reset = Exhaust
```

后续进入原框架复位收尾 `Step_JL_Reset()`。

原框架负责：

- 排气/复位收尾。
- 保存检测数据。
- 打印标签。
- 清理状态。
- 回到空闲。

## 6. 人工复位按钮逻辑

当前复位按钮已经显示。

点击复位按钮：

1. 主界面调用 `m_WorkTask->requestManualReset()`。
2. 主界面停止计时。
3. 主界面清显示。
4. 工作线程下一个循环检测到复位请求。
5. 工作线程调用 `performManualReset()`。

`performManualReset()` 执行：

- 清 `pendingBarcode/currentBarcode`。
- 清测试快照。
- 清 ATEQ 接收标志。
- 清 PLC 结果锁存。
- 清 ATEQ timeout。
- 清启动检漏标志。
- 发 `D1900.4` 全部复位。
- 停止计时。
- 清旧复位状态。
- 设置 `b_gameover = true`。
- 状态回到 `NE`。
- 清界面。

人工复位语义：

```text
人工中止整轮
不保存未完成数据
不打印未完成标签
发 D1900.4 全部复位
回到空闲
允许下一轮
```

## 7. 急停逻辑

急停显示控件已隐藏。

但工作线程仍保留急停保护：

- 急停状态下拒绝扫码。
- 急停状态下终止当前流程。
- 急停状态下切断相关输出。

注意：

当前急停输入仍来自原框架 `recDatasDi` 映射，界面不显示不等于取消安全保护。

## 8. 异常处理

### 8.1 扫码枪不可写

触发扫码枪时，如果串口：

- 未初始化
- 未打开
- 不可写

则：

- 不崩溃。
- 输出提示。
- 发 `sign_scanWriteFailed()`。
- 主界面扫码灯置红。

### 8.2 通道非法

通道判断：

```text
channel > 0 && channel <= 50
```

通道非法时：

- 不触发有效流程。
- 扫码灯置红。
- 日志记录。

### 8.3 ATEQ 未选择

当前合法 ATEQ 模式：

```text
1 = ATEQ1
2 = ATEQ2
3 = 双检
```

如果 ATEQ 为空或非法：

- 不进入检测。
- 扫码灯置红。
- 结果灯置红。

### 8.4 PLC 无结果

`JudgeResult` 阶段会等待当前 ATEQ 对应的 PLC 有结果位。

如果 PLC 一直不给结果，程序会停留等待。

### 8.5 ATEQ 串口超时

PLC 已给结果后，如果 10 秒内没有收到对应 ATEQ 串口数据：

- 日志输出：

```text
[ATEQ] wait data timeout enable=... ateq1=... ateq2=...
```

- 结果灯置红。
- 本轮结束。
- 清当前条码。

### 8.6 PLC ReadError

当前不再读 `D9101`，因此不会因为不存在的 `D9101` 产生读错误。

当前读取：

```text
D1901
D90
D91
```

若仍出现 `PLC ReadError`，应优先检查：

- PLC 逻辑站号。
- MX Component 连接。
- PLC 寄存器 D1901/D90/D91 是否可读。
- 通信是否断开。
- PLC 是否处于 RUN。

## 10. 当前主要风险点

### 10.1 扫码触发位仍不完美

当前代码仍用 `D1901.1` 上升沿触发扫码。

但按 PLC 表：

```text
D1901.1 = 整体检测状态
0 = 未检测
1 = 检测中
```

这意味着 `D1901.1` 不是严格意义上的“实体启动按钮按下，请求软件扫码”信号。

风险：

- PLC 进入检测中时可能被软件误认为扫码入口。
- 如果 PLC 状态时序变化快，可能出现重复触发或错触发。
- 如果扫码后软件又通过 `D1900.5` 启动检漏，PLC 侧状态设计必须能接受这个时序。

建议：

PLC 侧增加一个明确的 PLC -> 软件输入位，例如：

```text
启动按钮请求扫码
0 = 无请求
1 = 请求软件触发扫码枪
```

软件检测该位上升沿触发扫码。扫码完成并写好品番/使能后，再由软件置 `D1900.5` 启动检漏。

### 10.2 ATEQ2 单检显示位置

当前 `slot_updateAteqDisplay()` 中：

- 双检时 ATEQ2 显示到项目二。
- ATEQ2 单检时当前逻辑会允许显示，但 `secondItem` 默认仍是 false。

这意味着 ATEQ2 单检的压力/泄漏值可能显示到项目一位置。

如果业务要求：

```text
ATEQ1 永远显示项目一
ATEQ2 永远显示项目二
```

则需要把 `secondItem` 调整为：

```text
device == 2
```

### 10.3 BarTender 未安装或 ActiveX 不可用

日志中出现过：

```text
QAxBase.setControl: requested control BarTender.Application could not be instantiated
```

如果打印模块未完整做空指针保护，可能造成打印阶段异常。

建议：

- 初始化 BarTender 后判断 `isNull()`。
- 获取 `Formats`、`Open`、`PrintSetup` 前逐级判空。
- BarTender 不可用时跳过打印并记录日志，不能影响检测主流程。

### 10.4 保留了历史流程代码

项目中仍保留旧的：

- `Wdata["启动检漏"]`
- 旧复位流程变量
- 旧排气/对插/正负压通道代码

部分已经不是当前主链路核心，但仍在旧框架收尾中存在。后续若继续重构，应分阶段清理，不能一次删除，否则可能影响保存、打印、复位收尾。

## 11. 推荐最终稳定闭环

最稳定的最终闭环建议如下：

```text
PLC 实体启动按钮按下
    -> PLC 给软件一个独立启动请求位
    -> 软件检测该位上升沿
    -> 软件触发扫码枪持续等待条码
    -> 收到条码后校验
    -> 软件写 D1902/D1903
    -> 软件写 D1900.3/D1900.9
    -> 软件发 D1900.5 启动检漏
    -> PLC 开始检漏并置 D1901.1 检测中
    -> PLC 给 D1901.6/9 有结果
    -> 软件读取 D1901.7/10 结果
    -> 软件等待 ATEQ 串口压力/泄漏值
    -> 软件发 D1900.4 全部复位
    -> PLC 清结果/状态
    -> 软件保存/打印
    -> 软件回空闲
```

其中：

- `D1901.1` 只表示检测中。
- `D1900.5` 只表示软件启动检漏。
- `D1900.4` 只表示软件全部复位。
- 触发扫码必须使用一个 PLC -> 软件的独立输入位。

## 12. 当前测试重点

建议现场按以下顺序测试：

1. 打开软件，确认 PLC 无 `ReadError`。
2. 确认扫码枪串口打开。
3. 确认 ATEQ1 COM4、ATEQ2 COM2 打开。
4. 选择 ATEQ1，触发一次流程。
5. 确认条码显示到输入框。
6. 确认条码通过后 `D1900.5` 产生启动脉冲。
7. 确认 PLC 给 `D1901.6/7`。
8. 确认 COM4 收到 ATEQ 数据。
9. 确认 `D1900.4` 产生全部复位脉冲。
10. 确认保存/打印完成后回到空闲。
11. 选择 ATEQ2，重复验证 `D1901.9/10` 和 COM2。
12. 选择双检，确认两个结果和两个串口数据都到后才收尾。
13. 中途点击复位按钮，确认立即清状态并发 `D1900.4`。
14. 验证扫码枪未打开时不会崩溃。
15. 验证 ATEQ 串口未返回时 10 秒 timeout。

## 13. 结论

当前代码已经形成了“扫码、条码校验、品番写入、D1900.5 启动检漏、PLC 结果等待、ATEQ 数据等待、D1900.4 全部复位、保存打印、回空闲”的主体闭环。

但严格意义上，当前仍有一个关键待确认点：

```text
缺少独立 PLC -> 软件的实体启动按钮请求扫码输入位。
```

只要仍然借用 `D1901.1` 触发扫码，逻辑就存在状态语义混用风险。若 PLC 后续提供独立启动请求位，软件应替换当前 `D1901.1` 上升沿触发扫码逻辑，届时闭环会更稳定、更符合表格定义。
